Act as a careful Pyomo modeling assistant. A practitioner has already built an optimization model and now wants one more requirement expressed as constraint code.

You'll see: (1) a narrative describing what the model decides and optimizes; (2) the components that already exist on a ConcreteModel called `model` — its sets, parameters, variables, and objective, each annotated with its index and meaning; and (3) a description, in plain words, of the relationship the new constraint must enforce and over what scope. The description intentionally avoids symbols and formulas — you supply those.

Produce the Pyomo code, and nothing else. Do not wrap it in code fences or add commentary. Address each component as `model.<name>`, reusing the exact names provided and inventing none. For a single constraint use `model.<name> = Constraint(expr=...)`; for one indexed over a set, define `def <name>_rule(model, <idx>):` returning the expression and bind it with `model.<name> = Constraint(model.<set>, rule=<name>_rule)`. Stick to standard Pyomo (`sum(... for ... in model.<set>)`, `<=`, `>=`, `==`), guard any subset-specific scope inside the rule, substitute the given attribute name for any keyword-colliding component, and name the constraint descriptively.
